# HoundWhiteboard Electron 安全边界与 Safe I/O 约束

## 1. 总览

本文整理 HoundWhiteboard 在 Electron 架构下的安全边界设计，重点关注 renderer 进程与本地能力之间的隔离原则。

核心判断可以归纳为一句话：

- **对 renderer 保持零信任**

这里强调的不是“区块里没有暴露明显危险按钮，所以应用是安全的”，而是：只要 renderer 可被注入、篡改或伪造请求，它就不能被当成可信执行面。任何文件系统、系统调用、敏感 I/O、危险句柄，都不能直接交给 renderer 决策。

---

## 2. 问题定义

这里要解决的问题是：

- 为什么不能在 renderer 里直接承载高权限逻辑，甚至让它直接触达 Node 或后端能力

结论是：

- 攻击者可以在 renderer 世界中伪造看起来“合法”的请求
- 一旦 renderer 持有真实高权限能力，伪造请求就可能直接触发危险动作
- 因此安全设计的关键不是“前端写得多严谨”，而是“前端根本拿不到危险能力”

围绕这个问题，可以形成一组明确的防线：

1. 用沙箱和隔离机制阻止 renderer 越界访问
2. 把能力收敛成可审计的句柄或 handle，而不是暴露整块 Node 能力
3. 用类型约束、对象冻结、语言层限制等手段降低伪造与篡改空间
4. 通过 contextBridge 只暴露最小化、安全包装后的能力，不允许 renderer 自由生成访问路径
5. 对高风险调用增加显式申请、单一官方通道、运行时校验与动作级确认
6. 支持能力回收，避免授权长期悬挂

这几条合起来，构成的就是最小权限、显式授权、单向桥接、运行时校验四个核心原则。

---

## 3. 核心警告

最需要强调的警告是：

- 即使界面看起来是安全的，也不能把 renderer 本身视为安全
- 攻击者如果能改写 renderer 中的代码，就能伪造交互流程与参数
- 所以不能用“界面上没有这个入口”来证明“系统不可能执行这个动作”

这意味着安全验证必须落在真正持有能力的一侧，也就是 preload 或 main 对应的受控桥接层，而不是只靠 renderer 侧 UI 流程约束。

换句话说：

- UI 约束只能改善正常路径
- 权限约束和动作校验才真正决定攻击面

---

## 4. 对工程实现的直接要求

可以把工程要求整理成下面几条。

### 4.1 敏感 I/O 不进入 renderer

凡是涉及下列能力，都不应直接下放到 renderer：

- 文件读写
- 路径遍历
- 批量枚举目录
- 任意打开本地资源
- 具有副作用的系统操作

renderer 最多只能发出一个受限意图，例如：

- 读取当前白板文件的某个模板
- 请求保存指定 board 的当前快照
- 请求导出到受支持格式

真正的路径解析、权限检查、文件句柄管理、错误分类，应该在 preload 与 main 之间的受控链路里完成。

### 4.2 API 必须是白名单式的

不要给 renderer 一个“通用文件系统对象”或“万能 invoke 通道”，而应该提供非常具体的 API：

- `openBoard()`
- `saveBoard()`
- `readTemplate()`
- `writeChunkData()`

每个 API 都应有固定输入结构、固定输出结构、明确失败路径。

### 4.3 每个动作都需要运行时校验

这里要反复强调一点：即使请求来自“看起来合法”的区块流程，也仍然需要重新校验。

因此桥接层至少应检查：

- 参数类型是否合法
- 路径是否落在允许范围内
- 请求目标是否属于当前打开的 board 工作目录
- 动作是否属于当前能力令牌允许的范围
- 是否需要额外确认，例如覆盖写入、删除、导出

### 4.4 能力应可回收、可缩时

能力回收是必要设计，不应发放永久权限。

更稳妥的方式是：

- 为一次操作签发短生命周期 token
- token 只绑定单类动作或单个资源
- 操作结束立即失效
- 窗口切换、board 关闭、会话中断后统一回收

## 5. 与当前仓库的对应关系

从仓库结构看，这份笔记和当前工程已经有明显对应点：

- `src/preload-io.js`
- `src/io-bridge-common.js`
- `src/io-bridge-main.js`
- `src/core/bridges/file-operate-bridge-common.js`
- `src/core/bridges/file-operate-bridge-main.js`
- `src/core/bridges/file-operate-bridge-renderer.js`

这说明项目已经在往“桥接能力收口”的方向设计。这里的安全约束，本质上是在为这条路线做原则补充。

可以把它落实为下面几条仓库级约束：

1. renderer 不直接接触 Node 文件系统接口
2. 所有文件操作统一经由 bridge 层封装
3. bridge 层 API 不暴露原始路径拼接能力
4. main 侧做最终权限判断与路径归一化
5. 敏感动作必须可审计，至少要有明确调用点和错误分类

---

## 6. 可落地的 safe-io 方案

这里的关键方向是：

- 用 safe-io 或 safe-api 的方式，重建对敏感数据与本地能力的安全控制

结合当前项目，可以把 safe-io 理解为一层“最小化、安全包装、本地能力代理”框架。它至少应包含以下能力：

- 统一请求入口
- 参数 schema 校验
- 资源范围校验
- 能力令牌或会话级授权
- 明确的错误码
- 操作日志或调试日志

一个更贴近当前仓库的分层可以是：

1. renderer：只表达用户意图，不持有真实 I/O 能力
2. preload bridge：暴露受限 API，并做第一层参数约束
3. main bridge：执行真正 I/O，并做最终权限与路径校验
4. core/file layer：处理业务语义，不向上泄漏底层危险句柄

---

## 7. 开发纪律

除具体 API 设计外，还需要明确工程纪律：

- 在推进其他复杂任务前，优先把 safe-io 这类安全基础设施合并稳定
- 功能开发不能绕过既定桥接层直接下沉到 renderer
- 任何“先跑起来再补安全”的路径，后面都会放大修复成本

这部分对于当前仓库尤其重要，因为项目同时涉及：

- Electron 主进程
- preload 桥接
- 渲染层模板
- board 文件读写
- core 层区块与对象持久化

这些模块天然跨越多个信任边界。如果边界一开始没收紧，后面功能越多，攻击面和维护成本都会一起扩大。

---

## 8. 建议的后续动作

如果要把这些原则真正转成工程行动，建议按这个顺序推进：

1. 盘点当前 renderer 可直接触达的 I/O 能力与 Node 能力
2. 把现有 bridge API 列成白名单，并标出参数、返回值、权限边界
3. 在 main 侧补齐路径归一化、目录范围限制、动作校验
4. 为高风险操作设计短时授权或确认机制
5. 补一份 bridge 安全约束文档，作为后续开发基线

---

## 9. 一句话结论

这里的核心不是“Electron 很危险”，而是：

- Electron 必须用清晰的信任边界来设计
- renderer 只负责表达意图，不负责持有能力
- 真正安全的关键，在于 safe-io / safe-api / bridge 层是否把能力收口、校验、审计和回收做好

如果把这句话对齐到 HoundWhiteboard 当前工程，就是：先把 I/O 桥接做成受控系统，再继续叠加功能。
