# 文件粒度与 I/O 性能分析

本文档基于 benchmark/io-file-granularity.bench.js 的实测数据，分析在固定总数据量下，文件粒度对读写性能的影响。包括两组测试：小数据量（256 KB）和大数据量（81920 KB）。

## 测试设置

### 测试组 1：小数据量
- **总数据量**：1000 个原子单位，每个 256 字符的 JSON 对象
- **总大小**：约 256 KB
- **文件数范围**：1-1000 个

### 测试组 2：大数据量
- **总数据量**：10000 个原子单位，每个 8192 字符的 JSON 对象
- **总大小**：约 82 MB
- **文件数范围**：1-10000 个

### 通用测试设置
- **测试操作**：
  1. 读取：批量连续读取所有文件
  2. 修改：随机选取一个文件，随机修改其内容，写回
- **环境**：主进程 fs 模块（io.js 封装），Node.js

## 关键发现

### 1. 小数据量（256 KB）测试结果

#### 读取性能：小文件优势明显

- **最优配置**：500-1000 个文件（2-1 单位/文件），读取速度 **86,000-104,000 ops/sec**
- **性能悬崖**：单个文件超过 50 KB 时，读取速度开始明显下降
- **极限对比**：小文件场景（256 B）vs 单大文件（256 KB），性能相差 **6.3 倍**

#### 修改性能：中等粒度最优

- **最优配置**：50 个文件 × 20 单位（5,120 字符/文件），修改速度 **16,193 ops/sec**
- **次优范围**：20-100 个文件，修改速度在 11,000-16,000 ops/sec
- **极限对比**：最优配置 vs 单大文件，性能相差 **7.3 倍**

### 2. 大数据量（82 MB）测试结果

#### 读取性能：小文件优势依旧显著

- **最优配置**：10,000 个文件 × 1 单位（8,192 字符/文件），读取速度 **73,857 ops/sec**
- **次优范围**：5,000-10,000 个文件，读取速度 **69,000-73,000 ops/sec**
- **性能衰减**：文件大小超过 100 KB 时，读取速度急剧下降
- **极限对比**：最小粒度 vs 单大文件（82 MB），性能相差 **1,538 倍**

#### 修改性能：小数据量中等粒度原理不适用

- **关键发现**：大数据量下，修改单个数据项时，文件大小的影响远大于文件数量
- **次优配置**：5,000 个文件 × 2 单位（16,384 字符/文件），修改速度 **7,117 ops/sec**
- **极限对比**：最优配置 vs 单大文件（82 MB），性能相差 **844 倍**

### 3. 小 vs 大数据量对比

#### 读取性能差异

| 文件粒度 | 小数据量 (256 KB) | 大数据量 (82 MB) | 按文件数比 | 性能衰减 |
|---|---|---|---|---|
| ~8 KB/文件 | 102,735 ops/sec | 73,857 ops/sec | 1:10000 | **27.7%** |
| ~25 KB/文件 | 72,375 ops/sec | 39,199 ops/sec | 1:1000 | **45.8%** |
| ~256 KB/文件 | 16,206 ops/sec | 513.42 ops/sec | 1:100 | **96.8%** |

**结论**：单个文件大小相同时，数据量增加 320 倍，性能下降幅度从 27% 到 96%。小文件粒度对性能保持的影响显著。

#### 修改性能差异

| 文件粒度 | 小数据量 (256 KB) | 大数据量 (82 MB) | 按文件数比 | 性能衰减 |
|---|---|---|---|---|
| ~8 KB/文件 | 13,336 ops/sec | 7,770 ops/sec | 1:10000 | **41.7%** |
| ~81 KB/文件 | 13,888 ops/sec | 4,002 ops/sec | 1:1000 | **71.1%** |
| ~256 KB/文件 | 2,223 ops/sec | 9.16 ops/sec | 1:100 | **99.6%** |

**结论**：大数据量下，修改单个数据项的成本随文件大小非线性增长。

### 4. 文件系统与缓存的影响

#### 缓存命中率差异

- **小数据量**：总体系统缓存能容纳，即使是 256 KB 的文件也能被整体缓存
- **大数据量**：82 MB 的数据超出 L3 缓存（通常 8-20 MB），不同核心间缓存失效
- **文件数增加**：系统需要管理更多的文件描述符，inode 缓存作用减弱

#### 结论

大数据量时，小文件粒度相比中等粒度的优势**倍增**，原因包括：
1. 更小的块被加载到高速缓存中
2. 减少了单次寻址和解析的数据量
3. 充分利用多核 CPU 的 L1/L2 缓存

## 注意事项

1. **平台差异**：本测试在本地 SSD（MacBook Air M4 13 inch 2025 @ MacOS 26.4.1）运行。网络存储会放大小文件的开销（网络往返），应该采用更大的粒度（20-50 KB）。机械硬盘会放大寻址成本，小文件优势可能减弱。

2. **缓存影响**：
   - 小数据量（<256 MB）：大部分文件被 L3 缓存覆盖，性能对标记命中敏感
   - 大数据量（>1 GB）：缓存失效频繁，小文件粒度的优势被放大

3. **并发访问**：单进程情况下文件不会有锁竞争。多进程或 Web 服务模式下，小文件数量过多会增加同步复杂度，建议保持在 10,000 以内。

4. **JSON 序列化开销**：如果改用二进制格式（如 Protocol Buffers、MessagePack），JSON 解析成本消失，相对性能差异会收敛。

5. **大数据量下的特殊考虑**：
   - 当总容量超过 100 MB 时，单个文件超过 50 KB 会导致修改成本超线性增长
   - 对于频繁修改的数据，粒度应更细（8-16 KB）以获得最佳缓存效率
