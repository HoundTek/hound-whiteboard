# `.hwb` 文件结构文档

本文档提供 `.hwb` 文件的结构的概述。

## 文件读写之哲思

### 〇、为什么

为什么你的白板要存储起来？

乍一听，这就像是一句废话。我写在白板上的内容肯定要存储起来，这样下一次打开的时候就可以继续写。这就说到点子上了。我们把白板存储起来，就是想让用户关闭后打开看到的内容与关闭前一样。

### 一、存什么

那一个白板文件（.hwb）应该存什么，才能使用户关闭后打开看到的内容与关闭前一样呢？很简单，把它的内容存起来就行了。

那什么是白板的内容呢？内容包括：页信息，页关系，对象信息，对象关系，对象与页的关系，时间回溯树和上次关闭时的状态。

### 二、怎么存

我们使用压缩打包、解压拆包、基于文件夹树状结构的方式存储。关于对象的内容放在一个文件夹中，关于时间回溯树的内容放在一个文件夹中，对象与页的关系用文件夹实现……

但是，假设一个对象就是一个文件、一页就是一个文件，那文件夹中的小文件就太多了。于是我们采用文件块的方式整合小文件，避免读取慢的问题。

块有两种：有序块和无序块。无序块中块内与块间不保证有序；有序块中块内与块间都必须有序。也就是说，在有序块中，可以通过顺序读块的方式来顺序遍历小文件；在无序块中，想要顺序遍历，只能将所有块读完再排序。因此，无序块被用于对象、页、状态、垃圾、版本等的整合；有序块被用于时间回溯树中的树块等的整合。

## 概述

`.hwb` 文件是基于 `.zip` 文件的。在 Hound Whiteboard 打开一个白板时，`*.hwb` 文件会被解压到 `.*/` 目录下。

以下代码块提供了一个 `.hwb` 文件解压后的文件夹结构示例：

```
.hwb/
  devices/
  history/
    trash/
      page1/
        1.json
      page2/
        2.json
      pages/
        3.json
    edition/
      page1/
        2-1.json
        2-2.json
      pages/
        1-1.json
    hit/
      1.json
      2.json
  objects/
    stash/
      6.json
      7.json
    page1/
      3.json
      4.json
    page2/
      5.json
  pages/
    connection.json
    1.json
    2.json
  templates/
    1.json
    2.json
  config.json
  trace.json
  meta.json
```

## 内部结构

### 历史记录：`history/`

该文件夹用来存放所有对象和页的历史版本、已删除的对象和页，以及 hit 的块。

#### 回收站：`trash/`

该文件夹用来存放所有已被删除的对象和页。

#### 对象历史版本：`edition/`

该文件夹用来存放所有对象和页的历史版本

#### 时间回溯树：`hit/`

该文件夹用来存放 hit 的块。

### 对象：`objects/`

### 页面：`pages/`

该文件夹存放了关于页面的连接和对象层叠。

#### 连接顺序：`connection.json`

该文件存放了页的连接顺序。

示例：
```json
{
  "count": 8,
  "order": [1, 6, 3, 4, 2],
  "size": 5,
}
```

其中，`count` 是给 `CounterPool` 使用的。

#### 页层叠图：`{id}.json`

其文件名就是页 id，内部存放的是层叠图 DAG 数组化的结果。

### 白板配置：`config.json`

该文件存放了白板的基础配置。

### 上次打开：`trace.json`

该文件存放了上次关闭白板时的情况。

示例：
```json
{
  "onPage": 3,
  "offset": 0.1,
}
```

表示上次关闭时在第三页，但第三页的左边有 10% 没有显示。

### `meta.json`

白板的元数据。
