# HoundWhiteboard Core 总览

## 1. App 分层概览

当前工程可以粗分为三层：

1. Electron 主进程层
2. 模板/渲染层
3. Core 领域层

其中：

- 主进程入口在 `src/main.js`，负责窗口生命周期、IPC 注册、白板文件打开/保存调度。
- 业务管理组件在 `src/components/`（如板文件管理、模板管理、设置管理）。
- Core 在 `src/core/`，负责白板领域模型本身（页、对象、工具、层级关系、历史结构）。

## 2. 主流程（从启动到打开白板）

### 2.1 启动

应用启动后，主进程会初始化：

- 设置管理器
- 白板管理器（主进程版本）
- 模板管理器

随后创建主菜单窗口，并挂载 IPC 通道。

### 2.2 新建/打开白板

主进程侧 `board-manager` 负责：

- 创建 `.hwb` 结构（meta/config/pages/templates 等）
- 打开 `.hwb`（解压到临时目录）
- 保存白板（压缩回 `.hwb`）

这部分以文件系统与窗口切换为主，属于“宿主调度层”。

### 2.3 Core 介入点

Core 层中的 `BoardManager`（`src/core/components/board-manager.js`）是领域管理器，目标负责：

- 页生命周期与加载策略
- 对象增删改与层叠关系
- 活动对象状态
- Undo Tree 历史

当前代码中，该类骨架与主要字段已建立，部分流程仍在实现中。

## 3. Core 目录职责

- `components/`: 白板、页面、活动对象等管理器
- `objects/`: 各类领域对象（笔画、多边形、文本、容器）
- `tools/`: 工具体系（创建/修改/选择/擦除/白板工具）
- `hit/`: 历史与回溯树（Undo Tree）
- `devices/`: 输入、输出设备抽象
- `utils/`: 核心专用算法与数据结构（图、几何算法、计数池）
- `range/`: 选择与碰撞相关范围抽象
- `docs/`: 核心说明文档

## 4. 设计与实现的差异（现状）

当前 Core 特征：

- 设计文档较完整：
  - 工具栈与工具分类
  - 活动对象层叠图（静态图 + 动态层）
  - Undo Tree 与历史组织
  - `.hwb` 文件结构
- 实现完成度不均：
  - 已具备：`DirectedGraph`、基础对象体系、部分创建/修改工具、活动对象管理核心算法
  - 待完善：页对象持久化细节、Undo Tree 具体操作、选择/擦除/白板工具落地、多模块联调
